# tech-docs-guard 问题分类(重点检查清单)

> 目的:把 ops-nn《AI Core 算子开发指南》真体检暴露的问题**沉淀成可复用的检查类目**,
> 让 skill 后续**按类系统化排查**(finder 拿它当 checklist,逐项过)。
> 力求 **MECE(不重不漏)**:每条真实缺陷应**恰好**落进一个叶子类;每个叶子类都对应一种可独立判定的检查。
> 与五轴的关系:这是五轴(尤其「信得过」)的**操作化拆解**--五轴是「看什么」,本清单是「具体查哪几件事、怎么查」。

依据:2026-06-22 至 2026-06-23,基于 174+ 条「codex 已审」真实发现的四源实证,以及批1 311 篇全量深扫的 417 条发现、全量 601 条深扫频率证据。全量频率中 C6=177、C5=152、C7=88,C7 为第 3 大类;批1 C7=71 且全量 C7.1=85,因此 v3 恢复并细分 C7。全量深扫显示约 73.8% 文档存在某类问题,但其中大量问题对开发者低杀伤,若全部同权展开会造成 token 与修改负担过重;v4 新增正交 `impact` 维度,用于按开发者实际后果分层。实证中 12 个大类全部命中,无一发现落到分类外,因此本轮不新增大类;零命中叶子优先合并,不删除在其它大教程中仍可能触发的有效检查。

## 一、可量化检查类(对照代码 / grep 可确定性判 -> 可下「确认对不上」、可计数)

### C0 分诊与互斥落类规则(先判对象,再判缺陷)
检查前先问「这条缺陷的两个对照端是什么」,再落到唯一叶子类:
- **doc 链接文本 -> 仓内目标/锚点** -> C1。例:相对链接目标文件存在但锚点全文 0 命中。
- **doc 点名路径 -> 仓内文件/目录** -> C2。例:正文让参考的样例目录不存在。
- **doc -> 仓内真实定义/实现的名字** -> C3。例:文档头文件名 vs 仓内真实头文件名。
- **doc 内嵌代码片段 -> 片段自身或钉住的样例工程** -> C4。例:教程代码块与 `add_example` 样例骨架不一致。
- **doc -> doc 的不同陈述** -> C5。例:同一文档前后默认值、路径、术语说法互相冲突。
- **doc 参数表/接口说明 -> 算子定义、API 签名、schema、代码校验契约** -> C6。
- **doc 版本/SOC/分支/支持范围声明 -> 版本矩阵、代码条件、测试/发布证据** -> C7。
- **doc 命令执行后的输出/产物/安装路径声明 -> 构建脚本、CMake、install 规则、实际产物约定** -> C8。
- **doc 数值、公式、派生语义、注释行为 -> 可静态推出的代码行为** -> C9。
- **分诊排除规则**:占位符(`<x>` / `$VAR` / `{soc}`)、生成/安装产物(`build/` / `dist/` / `opp/vendors/...`)、外部命令(cmake/pip/git)不按字面判「路径缺失」。先判断它是变量、产物还是外部依赖;只有文档点名仓内静态位置时才进 C2。
- **C5/C6 边界(最高歧义)**:dtype/参数表/签名类问题,若**同篇文字之间矛盾、无需代码即可判错** -> C5;若**文档 vs 代码契约不一致,反证来自代码/API/schema/校验逻辑** -> C6。例:同篇两个接口的 dtype 表互相矛盾,且不需要代码裁决即可发现差异,归 C5.1;`aclnnElu` 参数表漏列代码支持的 `DOUBLE`,反证是 `DTYPE_SUPPORT_LIST` 与 `CheckDtypeValid`,归 C6.3。
- **优先规则**:若同一原文同时含多个问题,按最小可修复事实拆成多条 finding;若只能修一处就消失,只报那个直接事实错位对应的 category。不要把一个 finding 同时报 C3/C4/C5。

### C1 引用可达性(超链接 / 内链 / 锚点)
检查文档里的链接能不能到达、到达处对不对。
- **C1.1 链接目标存在** -- `[..](path)` 指向的文件/相对路径在仓内存在。
- **C1.2 锚点存在** -- `#anchor` 在目标文件里有对应标题(按渲染后 heading id 比对)。
- **C1.3 链接内容相关** -- 目标章节/锚文本与正文所述主题一致(非指向无关章节)。外部 URL 只查明显失效格式或协议错误;不可联网静态确证时不计入硬结论。
- 真实例:`[未联网编译](../invocation/quick_op_invocation.md#未联网编译)` 目标文件在但锚点全文 0 命中=死链 [#2/#5,CONFIRMED]。

### C2 路径 / 目录存在性(文档点名的仓内位置)
检查文档让你「创建 / 编辑 / 参考 / 进入」的具体仓内路径是否真实存在。
- **C2.1 文件路径存在** -- 举例或要求的具体文件。
- **C2.2 目录存在** -- 让你「参考 / 进入」的目录。
- 真实例:`add_example` 下无 `add_example_data` 目录,正文却让人参考 [#1/#10]。

### C3 命名 / 标识符一致(文档写的名字 vs 仓内真实定义)
只检查「名字本身」是否一字不差;不检查接口结构、命令产物、文档内部矛盾。
- **C3.1 文件 / 头文件名** -- 拼写、下划线、扩展名。
- **C3.2 命令 / 子命令 / flag 名** -- 如 `build.sh --genop`;只看命令/flag 是否存在,不看执行后产物(C8)。
- **C3.3 代码对象名 / 配置键 / 术语化代码名** -- 配置键、枚举项、函数、类、宏、类型、模板参数、命名空间,以及文档把仓内对象当专名引用时的大小写、前后缀。默认值是否对得上归 C6.3 或 C9。
- 真实例:头文件写 `infershape_context_faker.h`(无下划线),共享 common 里真名是 `infer_shape_context_faker.h`(有下划线)[#3]。

### C4 内嵌代码片段正确性(文档贴出的代码块本身)
检查文档里贴出的代码块、命令块、局部 diff、样例片段本身能不能作为该片段成立。若对照端是仓内真实 API/schema,改报 C6;若只是文档前后陈述打架,改报 C5。
- **C4.1 语法 / 拼写** -- 关键字、括号、分号,如 `#incldue`。
- **C4.2 片段内标识符自洽** -- 同一片段里声明与使用一致(声明 `inputGMX_` 却用 `inputGMX`)。
- **C4.3 与文档钉住的样例一致** -- include 风格、字段结构、调用形态不与同文明确要求参考的样例工程冲突。
- **C4.4 片段上下文完整性** -- 对标称「可复制」的片段,必要 include、命名空间、插入文件、前置变量齐全;示意片段只要求标明「示意/省略」。
- 真实例:`#incldue` 拼写错 [#11];`.h` 骨架声明带下划线、Init 里用无下划线,照抄编译失败 [#6]。

### C5 文档自洽性(陈述之间互相矛盾,无需外部代码也能判)
只检查文档内部不同陈述是否打架;不引入仓内代码、真实样例或运行行为作为反证。代码只能作辅助说明,不能成为 C5 的必要裁决依据。
- **C5.1 跨节同一对象一致** -- 同一路径、参数、命名、默认值、dtype 表在不同小节不一致。
- **C5.2 散文 vs 本文链接文字 / 本文样例说明一致** -- 正文说 A,紧随链接文字或样例说明说 B;若链接目标不存在归 C1,若样例代码与真实工程不一致归 C4/C6。
- **C5.3 同一流程阶段命名一致** -- 创建、编译、安装、卸载、调用各步骤对同一对象的命名规则一致;只凭文档文本即可看出冲突。
- 真实例:`vendor_name` 默认 `custom`、卸载却硬编码 `custom_nn`、aclnn 又写 `${vendor_name}_nn` [#7];`out` 输出行说明误写成 `upper`,同篇参数表即可看出复制错误。

### C6 接口 / schema / 参数表一致(doc 说明 vs 代码契约)
检查文档中的接口、参数表、算子定义说明是否与仓内可解析契约一致。它关心「结构和类型」,不是单个名字(C3)或片段语法(C4)。
- **C6.1 API 函数签名** -- 函数名、参数个数、顺序、类型、返回值、模板参数与声明/定义一致。
- **C6.2 算子定义 / schema** -- input/output/attr 名称、数量、required/optional、默认值与 op proto / json / yaml / register 代码一致。
- **C6.3 参数表字段语义** -- 文档表格里的取值范围、单位、默认值、枚举含义、dtype/format 支持集与 schema 或代码检查逻辑一致。
- **C6.4 示例输入输出契约** -- 示例 shape、rank、dtype、format、broadcast 规则与算子定义和校验逻辑一致。
- 判定例:若参数表写 attr 默认 `axis=0`,代码 schema 默认 `axis=-1`,归 C6.3;若 `aclnnEluGetWorkspaceSize` 的 self/out dtype 列漏列代码 `DTYPE_SUPPORT_LIST` 支持的 `DOUBLE`,归 C6.3;若两个接口的 dtype 表互相矛盾但是否支持 `DOUBLE` 需要代码共用列表裁定,最终仍按代码契约归 C6.3。

### C7 版本 / SOC / 分支 / 支持范围声明
检查文档对适配范围的声明是否有可对照证据;没有证据时标「未佐证声称」,有反证时才下「确认对不上」。
- **C7.1 产品支持表 SOC/型号 √× vs 代码注册** -- 产品支持表里的产品系列、芯片型号、√/×、漏列产品行,与 `AddConfig("ascend950")`、`AddConfig("ascend910_93")`、`socSupportMap`、`NpuArch::DAV_*`、per-SOC dtype 注册、`op_host/config/<soc>/`、CMake `SUPPORT_COMPUTE_UNIT`、README/aclnn 同算子支持表等证据一致。例:`quant/quantize/README.md` 把 Atlas 训练系列产品标 √,但 `aclnn_quantize.cpp` 的 `socSupportMap` 只覆盖 310P、910B、950 等 arch,旧 910/910A 未注册且会走 empty dtype map;`foreach_copy` 产品表把 Atlas A3 标 ×,但 `foreach_copy_def.cpp` 注册 `ascend910_93`,且仓内文档明确 Atlas A3 对应 `ascend910_93`,应改 √;多篇 foreach aclnn 文档漏列 Kirin X90/Kirin 9030,而 def.cpp 与 README 均列支持。
- **C7.2 CANN / torch_npu / 编译器版本声明** -- CANN、torch_npu、编译器、API 最低版本/推荐版本/引入版本、废弃版本等声明与仓内条件编译、头文件、发布说明、适配矩阵一致;批1未暴露高频硬例,但该叶保留用于后续跨版本教程审计。判定例:文档声称某 API 需 CANN 8.0 起,但头文件或发布说明显示 8.3 才引入,归 C7.2。
- **C7.3 分支 / tag / 目录版本一致** -- 文档声称适用的分支、tag、目录代际、V1/V2/V3/V4 接口范围不混用;同算子 README、aclnn 文档、V2 文档、示例头注释对同一版本对象的支持范围一致。例:`experimental/activation/fast_gelu_v2/docs/aclnnFastGeluV2.md` 写「Atlas A5 训练/推理系列」和 `Ascend950PR`,但仓内 A5 佐证只出现「Atlas A5 推理系列,DAV_3510」,且本算子 def 只注册 ascend950;应对齐目录/版本口径与产品术语,去掉无证据的「训练」声明。若 V1 不支持、V2 支持,文档必须显式说明接口版本差异,否则也归 C7.3。
- **C7.4 能力范围声明** -- 动态 shape、精度/误差、性能倍数、format、非连续 Tensor、TensorList 长度、FP8/HIF8/INT4、融合、确定性、是否仅 experimental、是否仅特定后端/SOC 已验证等能力范围声明,与 SOC 分支、代码守卫、dtype/format 门控、测试矩阵、benchmark、issue/PR、发布说明一致。例:`aclnnDynamicQuant.md` 的按产品 yOut 类型漏写 Ascend 950PR/950DT 的 HIFLOAT8/FLOAT8_E5M2/FLOAT8_E4M3FN,而代码这些输出类型只在 RegBase/950 分支返回;`foreach_mul_scalar/README.md` 把 x 最大 Tensor 个数 50 只放在 950 小节,但钉住的 proto 注释表明 list length 上限是通用约束,需核实后移到通用约束或补 SOC 限定。
- 判定纪律:产品支持表的 √/× 和 SOC 条件 dtype/能力常同时出现,优先按「声明对象」拆分:产品行对不对归 C7.1;CANN/torch_npu/编译器版本归 C7.2;分支/tag/目录/V1-V4 口径归 C7.3;同一产品下 dtype/FP8/shape/长度/精度/性能等能力细节归 C7.4。若反证只是参数表 dtype 与代码支持集不一致、没有 SOC/版本/能力范围维度,仍归 C6.3。

### C8 命令输出 / 产物路径 / 安装结果一致
检查教程让读者运行命令后「应该看到什么 / 生成什么 / 到哪里找」是否与脚本、CMake、install 规则一致。
- **C8.1 产物文件名 / 后缀** -- `.run`、`.so`、`.json`、`.om`、包名等与构建规则一致。
- **C8.2 产物目录 / 安装路径 / 清理结果** -- `build/`、`dist/`、`opp/vendors/...`、`output/`、clean/uninstall/re-run 后文件状态与实际生成、安装或清理规则一致;纯仓内静态路径缺失仍归 C2。
- **C8.3 命令期望输出** -- 成功日志、关键 stdout/stderr、退出码说明与脚本分支一致。
- 判定例:文档说 GE 图模式编译后在 `/build/bin` 下生成可执行文件,但 GE 段 CMake 未设 runtime output directory,run.sh 在 `build/` 下直接执行,归 C8.2;文档点名以 `test_aclnn_add_example` 为例却展示 `mean result[2046]`,真实程序只打印前 10 个 `add_example ... result[...] is:`,归 C8.3。

### C9 数值 / 公式 / 行为声称一致
检查文档里的可计算数值、公式、派生语义、注释行为是否与代码可静态推出的行为一致。
- **C9.1 常量 / 容量 / offset** -- UB 大小、block 数、tile 长度、对齐粒度等数值与代码常量或平台 API 返回值使用一致。
- **C9.2 公式 / 推导** -- tiling、shape、stride、workspace、GM/UB 搬运量等公式与实现一致。
- **C9.3 分支语义 / 枚举含义** -- 注释或正文解释的分支条件、枚举含义与代码控制流一致。
- **C9.4 示例结果值** -- 文档展示的计算结果、shape 推导结果、性能/精度数值与可复现证据一致;无执行证据只标「未佐证」。
- 真实例:注释称 ub=262144,实际代码为平台 UB 容量减 256 字节 [#12];注释把 `schMode` 解释成 dtype 开关但代码按 schedule mode 分支,归 C9.3。

## 二、不可量化判断类(LLM 阅读 + 先 steelman -> 最多「疑似 / 教学判断」)

### J1 概念 / 原理正确性
- **J1.1 概念没讲反 / 讲错** -- 原理、语义不被讲错(须外部反证才「确认讲错」,否则降「疑似概念风险」)。
- **J1.2 术语 / 同名消歧 / 概念边界** -- 一词多义或与框架同名物混淆(自定义 helper vs 框架 API;`schMode` 是「调度模式」还是「dtype 开关」);不同 CANN 版本、SOC、执行后端的概念差异是否讲清。若文档给出可核对支持声明,归 C7。
- 真实例:`GetPlatformInfo` 像框架 API 实为样例自定义 helper;若只有概念混淆风险归 J1.2,若代码注释语义可被控制流反证归 C9.3。

### J2 教学有效性
- **J2.1 讲清「为什么」与可迁移心智模型** -- 关键设计动机、取值依据、从本算子迁移到新算子的思路有交代(按教程类型 graded)。
- **J2.2 教学锚点一致** -- 正文骨架与所钉权威样例不在概念层冲突(单级 vs 两级切分模型)。

### J3 完整性 / 可操作(静态阻断;只报缺漏,**不下「能跑通」**)
- **J3.1 步骤完整 / 顺序正确** -- 必要步骤不缺、不颠倒。
- **J3.2 前置与环境交代完整** -- 环境变量、依赖、前提写明(如编译前需激活 CANN);涉及多版本或多 SOC 时写清选择规则、不匹配症状、替代路径。若声明本身可被反证,归 C7。
- **J3.3 指向与验证检查点明确** -- 「在 CMakeLists.txt 加...」说清是哪个文件/哪个分支;关键步骤说明期望产物、输出、下一步输入。若给出的产物路径或输出与脚本不符,归 C8。

## 三、正交维度(root_cause / impact)

`root_cause` 是 finding 的**可选标签**,与 `category` 正交。`category` 回答「错在哪个可检查面」,仍必须 MECE 地落到一个叶子类;`root_cause` 回答「为什么会错」,用于后续聚类修复、模板治理和回归预防。它不是新类目,不能替代 `category`,也不改变可量化/不可量化二分。

建议先保留少量高频根因标签:
- **copy_paste_not_updated(复制粘贴未改)**:批1+全量深扫实测第一根因,全量 601 条中约 321 条(≈53%)命中,批1 417 条中 270 条(≈65%)命中,远高于其它根因。常见形态包括同族算子支持表/参数表未按本算子改、README 与 aclnn 文档互抄后漏同步、示例注释/变量名沿用旧算子、目录树从模板复制后未按真实仓内结构修正。例:`aclnnRReluWithNoise` 参数表中 `out` 描述抄成 `upper`,category=C5.1;GE 图模式段把 aclnn 段的 `/build/bin` 产物路径照抄过来,但 GE 的 CMake/run.sh 实际在 `build/` 下生成并执行,category=C8.2;`op_api_list.md` 中 `LayerNormBackward` 行说明写成 `aclnnNorm` 的反向,category=C5.2;`foreach_copy`/`foreach_addcdiv_list` 把已有 `ascend910_93` 或 `ascend950` 配置的产品行标 ×,category=C7.1。
- **template_placeholder_left(占位模板未填/样例残留)**:模板行、占位字段、外算子对象名未替换干净。例:某些错误码表保留 `self/out`、`scaleOptional/shiftOptional` 等外算子残留对象,但真实接口参数不同,category=C6.3。
- **version_or_contract_drift(版本升级/代码契约演进漏改)**:代码支持集、分支逻辑或生成规则已变化,文档未同步。例:`aclnnEluGetWorkspaceSize` 参数表未补 `DOUBLE`,而代码 `DTYPE_SUPPORT_LIST` 已含 `DT_DOUBLE`,category=C6.3。
- **fabricated_template_tree(虚构目录结构/模板文档树)**:README 的「工程结构/目录结构/测试流程」来自模板或外部设计树,仓内实际没有这些文件、目录或脚本。它是 copy/paste 的反复子型,但可单独打标用于集中治理 README 目录树。例:`experimental/activation/hard_shrink_grad/README.md` 的目录结构列出不存在的 `docs/`、`probe/`、`tests/ut`、`tests/st` 与 `arch32/` 子树,category=C2.2;`experimental/activation/celu_v3/README.md` 写出本目录不存在的 `build.sh`、`docs/`、`tests/ut/run.sh` 和多组 CSV/benchmark 文件,category=C2.2/C8.3;`experimental/foreach/foreach_div_scalar/README.md` 画出不存在的 `docs/` 子树,category=C2.1。

批1未发现需新增的「写死私人绝对路径」根因标签:未见 `/home/cjl/...` 等私人路径高频样例。后续若出现,可新增 `private_absolute_path_left` 之类根因,但仍按 C2/C8/J3 等 category 落类。

### impact 开发者影响标签

`impact` 是 finding 的**可选标签**,与 `category`、`root_cause` 都正交。`category` 回答「错在哪个可检查面」,`root_cause` 回答「为什么会错」,`impact` 回答「开发者信了它会造成多大实际杀伤」。它不新增、不删除、不重排 C0-C9/J1-J3,也不改变 C5/C6 边界和可量化/不可量化二分。

`impact` 的判据是**后果**,不是类目;并且强依赖错误方向。同一 `category` 在不同方向上杀伤力可能相反,judge 必须按「后果 + 方向」定档,不能按 C 类/J 类或叶子号机械映射。

- **blocker(阻断/高)**:开发者照做会失败、编不过、跑不通、选到不能用的路径/接口/芯片/类型,或被卡在下一步。例:头文件名写错导致 `#include` 编不过;文档点名的目录/路径不存在导致步骤扑空;代码片段语法或片段内标识符不自洽导致编译失败;产品支持表假 √,文档标支持但代码没有注册,开发者据此选择后跑不了;dtype 表多列代码不支持的类型,开发者照用即被校验拒绝。
- **misleading(误导/中)**:会制造困惑、误判或额外排查成本,但开发者通常能通过跳转、上下文、报错或代码核对恢复。例:文件或锚点死链;同篇文档自相矛盾;示例运行输出贴错但不影响核心步骤执行;概念讲偏、术语边界没说清,导致理解成本上升。
- **minor(瑕疵/低)**:几乎不改变开发者行为或只造成轻微编辑/观感成本;即使信了也不太会失败。例:锚文本错字但链接仍可跳,如「互推关系」漏写「导」字;HTML `rowspan` 多算但表格含义仍能读;散文里大小写/风格不一致,如 `output_mask` vs `outputMask`,但没有作为代码标识符要求照抄;外部 URL 只是格式风格问题且不可静态证明失效;尤其是「欠声明」类,如 dtype/支持表漏列代码其实支持的能力,开发者顶多少用一个可用能力,不会因照做失败。

方向性样例:
- C6.3 dtype 表**多列代码不支持的类型** -> `blocker`:开发者按表使用会被代码契约拒绝。
- C6.3 dtype 表**漏列代码支持的类型** -> `minor`:文档少声明一个可用能力,开发者只是少用,不阻断既有步骤。
- C7.1 产品支持表把**未注册芯片标 √** -> `blocker`:开发者选到实际跑不了的芯片。
- C7.1 产品支持表把**已注册芯片标 × 或漏列** -> 通常 `minor`:开发者保守地不用一个其实可用的芯片;除非教程目标明确要求该芯片路线,才可升为 `misleading`。
- C1.2 锚点不存在 -> 通常 `misleading`:读者找不到目标章节但可用搜索或上下文恢复;若该链接是唯一下载/唯一下一步入口,可升为 `blocker`。
- C3.1 头文件名错误 -> 通常 `blocker`:它直接进入 `#include` 或文件创建步骤时会编不过;若只是散文提及且上下文有正确代码,可降为 `misleading`。

默认降为 `minor` 的清单:
- 链接锚文本、标题文字、散文说明中的错别字,只要链接目标仍可达且不改变操作对象。
- HTML/Markdown 展示层瑕疵,如 `rowspan` 多算、列表缩进轻微异常、表格视觉但语义仍可读。
- 散文里的大小写、下划线/驼峰、风格不一致,只要不是要求照抄的代码标识符、路径、命令或配置键。
- 外部 URL 的格式/风格问题;不能联网静态确证失效时,不要为低价值问题深挖。
- 「欠声明」类支持范围问题:代码支持但文档漏列 dtype、format、SOC、能力项,默认按 `minor` 处理;只有它造成读者误选流程、误判适配范围或放弃教程核心目标时才升为 `misleading`。

finder 策略:
- 对默认 `minor` 清单应短路:不深挖、不跨文件扩证、不做长链取证;记录最小事实即可,必要时直接折叠。
- 报告、排序与频率统计默认突出 `blocker` + `misleading`;`minor` 默认折叠收起,只在附录、明细表或用户明确要求全量时展开。
- 同一批扫描的资源预算优先给 `blocker` 和高置信 `misleading`;不要为了证明低杀伤错字、排版、欠声明问题消耗与阻断问题相同的 token。

用法约定:
- **优先级**:finder 先按 C0 分诊,再把 C1-C9(高置信、可计数)逐项过一遍,最后做 J1-J3(教学判断)。
- **判定纪律不变**:C 类走证据三态(确认 / 疑似 / 未找到)+ 工具不自下 CONFIRMED;J 类先 steelman 再 flag。
- **落库**:每条 finding 必填 `category`(本清单叶子类 id,如 `C1.2`),可选 `root_cause`(如 `copy_paste_not_updated`),可选 `impact`(`blocker` / `misleading` / `minor`)。judge 定 `impact` 时只看「后果 + 方向」,不要按 category 自动推断。
- **MECE 自检**:新缺陷若两个叶子类都像 -> 先按 C0 的「对照端」重判;若只是成因相同,放进 `root_cause`,不要新增或混用 category;仍无法唯一落类,说明边界不清,需调整本清单(而非硬塞)。
- **拆分纪律**:一句原文可以产生多条 finding,但每条 finding 只修一个事实错位。例如同一段同时写错 flag 名(C3.2)和写错产物目录(C8.2),必须拆两条。
- **证据门槛**:C3/C6/C8/C9 要优先找定义处或生成规则;grep 命中注释/旧文档只能作线索。C7 支持范围类若没有强反证,写「未佐证声称」,不要升级为「确认对不上」。
